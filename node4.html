Middleware — une fonction entre requete et reponse.

Un middleware recoit req, res et next. Il peut modifier req/res, terminer la requete, ou passer la main avec next(). Les middlewares s'enchainent dans l'ordre de declaration.

L'ordre compteExpress execute les middlewares dans l'ordre. Middleware global avant les routes. Un middleware sans next() ou res.json() bloque la requete indefiniment.
Express — Middlewares
const express = require('express');
const app = express();

// Logger — s'execute sur TOUTES les routes
app.use((req, res, next) => {
  console.log(req.method, req.url);
  next(); // Passer au suivant !
});

// Parser JSON
app.use(express.json());

// CORS
app.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin', '*');
  res.header('Access-Control-Allow-Methods', 'GET,POST,PUT,DELETE');
  res.header('Access-Control-Allow-Headers', 'Content-Type,Authorization');
  if (req.method === 'OPTIONS') return res.sendStatus(200);
  next();
});

// Routes (apres les middlewares)
app.get('/users', (req, res) => res.json([]));

// Middleware d'erreur (4 params — toujours en dernier)
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ error: err.message });
});

Middleware d'auth — proteger les routes.

Toutes les routes ne sont pas publiques. Le middleware auth verifie le token JWT avant de laisser passer. Sinon il renvoie 401 Unauthorized.

req.userUn middleware peut ajouter des proprietes a req. Le middleware auth decode le JWT et met req.user = { id, email, role }. Les routes suivantes lisent req.user sans re-decoder. Pattern standard — le middleware fait le travail lourd, les routes utilisent le resultat.
Middleware auth JWT
// middleware/auth.js
const jwt = require('jsonwebtoken');

const auth = (req, res, next) => {
  const header = req.headers.authorization;

  if (!header?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Token manquant' });
  }

  const token = header.split(' ')[1];

  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch {
    res.status(401).json({ error: 'Token invalide' });
  }
};

module.exports = auth;

// Dans les routes :
const auth = require('./middleware/auth');

// Route protegee — auth s'execute avant le handler
app.get('/profile', auth, (req, res) => {
  res.json({ user: req.user });
});

// Route publique
app.post('/login', (req, res) => { /* ... */ });

express.Router — organiser les routes en modules.

Quand l'app grandit, tout dans index.js c'est ingerable. Router groupe les routes par domaine dans des fichiers separes.

Structure recommandeeindex.js importe les routers. routes/users.js les routes /users. routes/auth.js les routes /auth. routes/portfolio.js les routes /portfolio. Chaque router est independant. C'est la structure de tout projet Express serieux — commence comme ca des le debut.
express.Router
// routes/users.js
const router = require('express').Router();
const auth = require('../middleware/auth');

router.get('/', auth, (req, res) => res.json([]));
router.get('/:id', auth, (req, res) => res.json({ id: req.params.id }));
router.post('/', (req, res) => res.status(201).json({ created: true }));

module.exports = router;

// routes/portfolio.js
const router = require('express').Router();
router.get('/', (req, res) => res.json({ portfolio: [] }));
module.exports = router;

// index.js
const express = require('express');
const app = express();
app.use(express.json());

// Monter les routers
app.use('/users', require('./routes/users'));
app.use('/portfolio', require('./routes/portfolio'));
app.use('/auth', require('./routes/auth'));

app.listen(3000);
📐
Controllers
Les routes gerent l'URL. Les controllers gerent la logique. router.get('/users', userController.getAll). Ca permet de tester les controllers independamment. Et si tu changes Express pour Fastify, tu gardes tous les controllers.
