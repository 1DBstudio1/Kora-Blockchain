PostgreSQL — la BDD de production.

SQLite est parfait pour dev. En production avec plusieurs utilisateurs simultanees, PostgreSQL est necessaire. Meme SQL, migration facile avec Prisma.

ElephantSQL ou SupabaseElephantSQL offre 20MB gratuit. Supabase offre 500MB + auth + storage + realtime — tout gratuit. Pour Kora proto : Supabase parfait. Railway propose PostgreSQL dans le meme projet que Node.js.
Migration SQLite vers PostgreSQL
# schema.prisma — changer le provider seulement

# Dev (SQLite)
datasource db {
  provider = "sqlite"
  url      = "file:./kora.db"
}

# Prod (PostgreSQL)
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

# DATABASE_URL format
# postgresql://user:password@host:5432/database

# Sur Railway — ajouter plugin PostgreSQL
# DATABASE_URL est injecte automatiquement

# Meme schema, meme client Prisma
# Meme code Node.js — zero changement

# Commandes
# dev :  npx prisma migrate dev
# prod : npx prisma migrate deploy

Supabase — le BaaS
Supabase = PostgreSQL + API auto-generee + Auth + Storage + Realtime. Tu crees une table dans l'UI et tu as une API REST instantanement. Pour Kora : Supabase pour prototyper vite avant de passer a Node.js custom.
Variables d'env en production — securiser les secrets.

Ne jamais hardcoder les secrets. Mots de passe BDD, JWT secret, cles API — tout dans des variables d'environnement.

Generer un JWT secret fortAu moins 32 caracteres, aleatoire. Depuis Node.js : require('crypto').randomBytes(64).toString('hex'). Ce secret ne doit jamais etre dans le code. Si compromis : change-le et tous les tokens deviennent invalides.
Variables d'env
# .env (jamais committe)
NODE_ENV=development
PORT=3000
DATABASE_URL=file:./kora.db
JWT_SECRET=un-secret-tres-long-minimum-32-chars
CORS_ORIGIN=http://localhost:8080

# Validation au demarrage
if (process.env.NODE_ENV !== 'production') {
  require('dotenv').config();
}

const required = ['JWT_SECRET', 'DATABASE_URL'];
required.forEach(key => {
  if (!process.env[key]) {
    console.error('Variable manquante:', key);
    process.exit(1); // Ne pas demarrer
  }
});

// Generer un secret fort
const crypto = require('crypto');
console.log(crypto.randomBytes(64).toString('hex'));
// Copie ce resultat dans .env et dans Railway

Gestionnaires de secrets
En prod serieuse : AWS Secrets Manager, HashiCorp Vault ou Doppler. Ces outils gere la rotation des secrets, l'acces par role, l'audit. Pour Kora au debut : variables Railway suffisent. Pour une vraie entreprise avec des donnees sensibles : vault dedie.
Securite basique — protections minimales.

Avant de partager Kora avec des proches : helmet, rate limiting, validation. Ce sont les protections minimales pour toute API publique.

npm auditnpm audit analyse les vulnerabilites dans les dependances. npm audit fix corrige automatiquement ce qui peut l'etre. A faire avant chaque deploiement. GitHub Dependabot envoie des alertes automatiques.
Securite — Protection basique
// npm install helmet express-rate-limit

const helmet = require('helmet');
const rateLimit = require('express-rate-limit');

// Securiser les headers HTTP
app.use(helmet());

// Rate limiting sur l'auth
app.use('/api/auth', rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 10,
  message: { error: 'Trop de tentatives, reessaie dans 15 minutes' }
}));

// Rate limiting global
app.use('/api', rateLimit({
  windowMs: 60 * 1000, // 1 minute
  max: 100
}));

// Validation avec Zod (npm install zod)
const { z } = require('zod');
const registerSchema = z.object({
  nom: z.string().min(2).max(50),
  email: z.string().email(),
  password: z.string().min(8)
});

router.post('/register', async (req, res) => {
  const parsed = registerSchema.safeParse(req.body);
  if (!parsed.success) {
    return res.status(400).json({ error: parsed.error.issues });
  }
  // ...
});

Logs de production
En prod utilise winston ou pino pour les logs structures : level (info/warn/error), timestamp, message. Ces logs vont dans un service comme Datadog ou Logtail. En dev console.log suffit. En prod les logs doivent etre persistent, searchable, et alerter sur les erreurs critiques.
