Profils publics — la vitrine des competences.

Chaque user a un profil public avec certifications, progression, KRT. Accessible sans connexion. C'est ce qu'on partage avec un recruteur.

Ne jamais exposer les donnees sensiblesProfil public : nom, pseudo, krt, certifications, formations, date inscription. Jamais : mot de passe, email (sauf si l'user le veut), tokens d'auth, adresses. Toujours separer donnees privees et publiques dans tes queries.
GET /users/:id/profile
// Public — sans auth
router.get('/:id/profile', (req, res) => {
  const user = db.prepare(
    'SELECT id, nom, krt, created_at FROM users WHERE id=?'
  ).get(req.params.id);

  if (!user) return res.status(404).json({ error: 'Introuvable' });

  const certs = db.prepare(
    'SELECT module_id, score FROM certifications WHERE user_id=? ORDER BY created_at DESC'
  ).all(req.params.id);

  const stats = db.prepare(
    'SELECT COUNT(*) as total, SUM(CASE WHEN statut="done" THEN 1 ELSE 0 END) as done FROM progressions WHERE user_id=?'
  ).get(req.params.id);

  res.json({
    id: user.id,
    nom: user.nom,
    krt: user.krt,
    membre_depuis: user.created_at,
    certifications: certs,
    stats
  });
});

Slug vs ID
GET /users/42/profile vs GET /users/valentin-dubromez/profile. Les slugs sont meilleurs pour le SEO et le partage. Stocke le slug en BDD (colonne slug UNIQUE). La librairie slugify genere proprement depuis le nom.
Feed d'activite — ce qui se passe sur Kora.

Le feed montre les activites recentes : certifications obtenues, modules termines, KRT milestone. Ca cree du dynamisme et encourage la communaute.

Table d'evenementsTable events : type (certification, module_complete, krt_milestone), user_id, metadata JSON, created_at. Le feed = SELECT sur cette table ordonne par date. Ajouter un nouveau type = ajouter un INSERT, pas changer le schema.
Feed activite
-- Table events
CREATE TABLE IF NOT EXISTS events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,
  type TEXT NOT NULL,
  metadata TEXT, -- JSON
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Logger un evenement
function logEvent(userId, type, metadata = {}) {
  db.prepare('INSERT INTO events (user_id,type,metadata) VALUES(?,?,?)')
    .run(userId, type, JSON.stringify(metadata));
}

// GET /api/feed
router.get('/', (req, res) => {
  const { limit = 20, page = 1 } = req.query;
  const events = db.prepare(`
    SELECT e.*, u.nom
    FROM events e JOIN users u ON e.user_id = u.id
    ORDER BY e.created_at DESC
    LIMIT ? OFFSET ?
  `).all(limit, (page-1)*limit);

  res.json({ events: events.map(e => ({ ...e, metadata: JSON.parse(e.metadata||'{}') })) });
});

// Apres certification
logEvent(userId, 'certification_obtained', { moduleId:'js12', score:18 });

WebSockets pour le temps reel
Le feed actuel necessite de recharger. Pour du vrai temps reel : WebSockets avec socket.io. Le serveur pushe les evenements des qu'ils arrivent. Chat, notifications live, prix crypto — tout ca passe par WebSockets. Module bonus apres l'API REST.
Railway — deployer en 5 minutes.

Railway est la plateforme la plus simple pour Node.js. Tu connectes ton repo GitHub, Railway detecte Node.js, installe et lance. Plan gratuit disponible.

process.env.PORTRailway injecte le port via process.env.PORT. Toujours const PORT = process.env.PORT || 3000. Si tu hardcodes 3000, ton app ne demarre pas. C'est l'erreur classique du premier deploiement.

  require('dotenv').config();
}

// CORS strict en prod
app.use(cors({
  origin: process.env.CORS_ORIGIN || 'http://localhost:8080'
}));

// Middleware d'erreur — pas exposer les details en prod
app.use((err, req, res, next) => {
  console.error(err);
  res.status(500).json({
    error: process.env.NODE_ENV === 'production'
      ? 'Erreur serveur'
      : err.message
  });
});

Render — alternative gratuite
Render.com plan gratuit pour Node.js. Endort l'app apres 15min inactivite (cold start 30s). Railway garde l'app active. Pour un prototype : Render gratuit suffit. Pour la prod : Railway payant ou VPS OVH.
