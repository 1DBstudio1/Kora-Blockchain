JWT — un token signe numeriquement.

Un JWT est une string en trois parties : header.payload.signature. Le payload contient les donnees (userId, role). La signature verifie l'integrite. Le serveur verifie sans BDD.

JWT vs SessionsSessions = un ID en BDD, pas scalable (un seul serveur). JWT = tout dans le token, stateless, scalable. Pour Kora : JWT pour l'API. Tokens courts (15min) + refresh tokens (30j) pour la securite optimale.
JWT — Structure
// npm install jsonwebtoken bcryptjs

const jwt = require('jsonwebtoken');
const bcrypt = require('bcryptjs');
const SECRET = process.env.JWT_SECRET;

// Creer un token
function creerToken(userId, role) {
  return jwt.sign(
    { userId, role },     // Payload
    SECRET,               // Secret
    { expiresIn: '7d' }   // Expiration
  );
}

🔑
Ne pas stocker JWT dans localStorage
localStorage est accessible par n'importe quel JS (risque XSS). Cookie HttpOnly non accessible en JS — meilleur choix. Pour une API mobile : localStorage acceptable si XSS securise. Pour Kora web : cookies HttpOnly.
Register et Login — les deux routes fondamentales.

Register cree un compte avec mot de passe hache. Login verifie les credentials et retourne un JWT.

Securite mot de passeNe jamais stocker en clair. bcryptjs hash avec un salt aleatoire. Le hash est irreversible. Pour verifier : bcrypt.compare(pwd, hash). bcrypt est intentionnellement lent pour ralentir le brute force.
Routes auth
const router = require('express').Router();
const bcrypt = require('bcryptjs');
const jwt = require('jsonwebtoken');

// POST /auth/register
router.post('/register', async (req, res) => {
  try {
    const { nom, email, password } = req.body;
    if (!nom || !email || !password)
      return res.status(400).json({ error: 'Champs manquants' });
    if (password.length < 8)
      return res.status(400).json({ error: 'Mdp trop court' });

    const existe = trouverParEmail(email);
    if (existe) return res.status(409).json({ error: 'Email utilise' });

    const hash = await bcrypt.hash(password, 12);
    const user = creerUser(nom, email, hash);
    const token = jwt.sign({ userId: user.id }, process.env.JWT_SECRET, { expiresIn: '30d' });

    res.status(201).json({ token, user: { id: user.id, nom, email } });
  } catch (err) { res.status(500).json({ error: err.message }); }
});

// POST /auth/login
router.post('/login', async (req, res) => {
  const { email, password } = req.body;
  const user = trouverParEmail(email);
  if (!user) return res.status(401).json({ error: 'Credentials invalides' });

  const ok = await bcrypt.compare(password, user.password_hash);
  if (!ok) return res.status(401).json({ error: 'Credentials invalides' });

  const token = jwt.sign({ userId: user.id }, process.env.JWT_SECRET, { expiresIn: '30d' });
  res.json({ token, user: { id: user.id, nom: user.nom } });
});

module.exports = router;
⏰
Refresh tokens
Access token : expiration courte (15min-1h). Refresh token : expiration longue (30j), stocke en BDD (revocable). Client utilise l'access token. Quand expire, envoie le refresh token pour en obtenir un nouveau sans reconnexion.
Routes protegees — /me et beyond.

Avec le middleware auth, n'importe quelle route accede a req.user. La route /me retourne le profil sans que l'user precise son ID.

req.user partoutUne fois l'auth execute, req.user est disponible dans tous les handlers. req.user.userId, req.user.role. Tu ne passes jamais l'ID dans l'URL ou le body — le token contient cette info. Plus securise : un user ne peut pas acceder aux donnees d'un autre en changeant l'ID.
Routes /me
// GET /me
router.get('/', auth, (req, res) => {
  const user = getUserById(req.user.userId);
  if (!user) return res.status(404).json({ error: 'Introuvable' });
  res.json({
    id: user.id,
    nom: user.nom,
    email: user.email,
    krt: user.krt
  });
});

// PUT /me — modifier son profil
router.put('/', auth, async (req, res) => {
  const { nom } = req.body;
  const updated = updateUser(req.user.userId, { nom });
  res.json(updated);
});

// Middleware admin
const adminOnly = (req, res, next) => {
  if (req.user.role !== 'admin') {
    return res.status(403).json({ error: 'Acces refuse' });
  }
  next();
};

// Route admin
app.get('/admin/users', auth, adminOnly, (req, res) => {
  res.json(getAllUsers());
});
🎯
401 vs 403
401 = pas authentifie. Client doit se connecter. 403 = authentifie mais pas les droits. 401 = 'Qui etes-vous ?'. 403 = 'Je vous connais mais non.' Client sait quoi faire : redirect vers login (401) ou afficher Acces refuse (403).
